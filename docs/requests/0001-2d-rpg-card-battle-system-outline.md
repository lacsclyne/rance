# 需求 0001：2D RPG 对战卡牌游戏系统结构大纲

## 背景

项目 `rance` 计划开始制作一个 2D RPG 对战卡牌游戏。第一阶段不实现代码，先形成一份系统结构大纲文档，用来统一后续玩法、数据、战斗、内容制作和技术架构方向。

## 目标

创建一份系统结构大纲文档：

```text
docs/system-architecture-outline.md
```

这份文档需要让后续开发者或 Codex agent 可以基于它继续拆分任务、设计数据结构、实现原型。

## 文档必须覆盖

1. 游戏定位

- 游戏类型：2D RPG + 对战卡牌
- 核心体验：角色成长、卡组构筑、回合制或半回合制战斗、关卡/敌人推进
- 第一版原型范围和暂不实现内容

2. 核心玩法循环

- 玩家进入关卡或战斗
- 选择角色/队伍/卡组
- 战斗中抽牌、出牌、结算行动
- 获得奖励、升级、解锁卡牌或装备
- 回到准备界面继续构筑

3. 主要系统模块

- 战斗系统
- 卡牌系统
- 角色与属性系统
- 敌人与关卡系统
- 成长与奖励系统
- 资源与经济系统
- UI/UX 页面结构
- 存档与配置系统
- 内容数据管理系统

4. 战斗系统大纲

- 回合流程
- 行动点/能量/费用机制
- 抽牌、弃牌、手牌、牌库、墓地
- 攻击、防御、治疗、状态效果
- 敌我双方行动顺序
- 胜负条件

5. 卡牌系统大纲

- 卡牌类型
- 卡牌稀有度
- 卡牌费用
- 卡牌效果表达方式
- 卡牌升级或强化方向
- 卡牌和角色职业/属性的关系

6. 数据结构初稿

用 Markdown 表格或 TypeScript-like pseudo schema 描述关键数据：

- `Card`
- `Character`
- `Enemy`
- `BattleState`
- `Deck`
- `StatusEffect`
- `Stage`
- `Reward`

7. 技术架构建议

- 推荐前端技术栈
- 2D 渲染方案
- 状态管理方式
- 数据驱动内容配置方式
- 测试策略
- 后续如果扩展为多人对战，需要预留哪些边界

8. 后续任务拆分

文档末尾列出 8 到 12 个后续 Linear issue 候选任务，例如：

- 初始化项目脚手架
- 实现战斗状态模型
- 实现卡牌数据 schema
- 实现最小战斗原型
- 实现基础 UI 页面

## 验收标准

- 新增 `docs/system-architecture-outline.md`
- 文档结构清晰，能作为后续开发任务的依据
- 不需要实现游戏代码
- 不需要引入依赖
- README 可按需增加一行指向该文档
- 修改完成后提交并推送到 GitHub

## 限制

- 不要把需求文档写成营销文案
- 不要直接实现游戏功能
- 不要删除现有 README
- 如果发现需求不明确，先在文档中明确假设，再继续给出结构方案
