# Rance 10 Card Design Reference

> 本文用于 `rance` 项目的系统设计参考。重点是拆解《兰斯10 决战》如何把大战略 RPG、角色收集、势力编成和回合制战斗连接起来；不复刻原作专有角色、剧情、图片或完整数据表。

## 调研边界

- 研究对象：《兰斯10 决战》的公开页面、官方系统说明、官方支持页、公开 wiki 与攻略资料。
- 记录方式：只总结系统结构和可迁移设计，不摘录长段原文，不搬运角色表、技能表、剧情表或图片。
- 事实可信度标注：
  - **官方事实**：来自 Alicesoft 官方页面。
  - **公开资料整理**：来自玩家 wiki / 攻略站，应作为设计参考而非唯一规格。
  - **项目启发**：面向 `rance` 项目的抽象设计建议。

## 原作事实 / 公开资料整理

### 1. 卡牌在系统中的定位

《兰斯10》不是传统构筑式卡牌游戏。卡牌更接近“角色/资产的数据库单位”，用于把大量人物、道具和奖励统一放入队伍、成长、战斗与收集系统。

官方系统页说明，宝箱主要产出角色，也可能产出道具、勋章、食券等对象；宝箱通常从若干张卡中选择一张，部分宝箱带固定候选或高稀有候选。由此可以拆成三个功能层级：

- **角色卡**：核心卡牌。角色被归入势力/所属组；可成为某组领队；领队提供可主动操作的战斗技能；非领队成员也计入该组战力。
- **道具/装备类卡**：官方说明道具也属于某个团队，能提高队伍战力；道具效果可通过地图对象强化。
- **奖励/进度类卡或卡状奖励**：勋章、食券、EXP、战果等不一定都是“角色卡”，但与卡牌奖励池同屏出现或同一地图奖励体系结算，承担部队升级、角色事件、全局战况等作用。

设计重点不是“每张卡都被打出”，而是让收集到的对象持续影响：当前队伍强度、可用技能、所属人数、部队等级、剧情事件、图鉴与周回目标。

### 2. 角色卡字段

公开资料中的角色卡字段可抽象为：

| 字段 | 原作中的作用 | 对系统的含义 |
| --- | --- | --- |
| 名称 / 个体身份 | 同名角色可能存在不同等级或特殊版本卡 | 角色身份与卡牌实例需要分层 |
| 所属 / 势力 | 主人公、国家、异族、怪物、神魔等组别 | 直接决定编成 tab、队伍合计战力、部分任务优势 |
| 星级 / 等级 / rank | 卡面上的星号数字代表 rank；EXP 使 rank 提升 | 是长期成长主轴 |
| HP / AT | HP 参与全队耐久，AT 参与所属队攻击 | 数值既属于个卡，也汇总到组 |
| 技能 | 领队在战斗中可用的主动/被动能力 | 决定该卡是否适合上场 |
| 属性 / 弱点适配 | 公开攻略按火、冰、雷、光、暗等弱点组织战斗 | 让战斗不是单纯堆数值 |
| 稀有/获取层级 | 官方区分普通宝箱、稀有宝箱、高稀有候选；公开 wiki 记录固定与特殊条件卡 | 更像掉落池权重和条件，而非 TCG 稀有度 |
| 重复获得 | 官方说明同一角色/同一卡重复获得会给能力 bonus | 让重复卡不是废卡 |
| 事件/食券条件 | 食券可触发角色事件并给 EXP；部分公开资料记录事件解锁卡 | 把收集、剧情、成长连接起来 |

一个重要观察：原作把“角色身份”和“卡牌版本”都放入卡系统。比如同一角色可能有不同等级、不同解锁条件或特殊版本。对本项目而言，建议拆分 `Character` 与 `CardVariant`，避免后期在一张卡上混合身份、成长、剧情状态和掉落条件。

### 3. 势力 / 阵营分类

官方系统页列出 10 个队伍/所属组：主人公、リーザス、ヘルマン、ゼス、自由都市、JAPAN、その他、亜人、モンスター、神魔。每个组是编成 tab，也是战力汇总单位。

势力的系统作用有四层：

- **编成限制**：每个前线配置位放置一个领队，相当于把该领队所属组派到前线。初期只开放部分配置位，强力的神魔组也需要部队奖励解锁。
- **战力汇总**：某组战力不是只看领队，而是该组已收集成员总和加上领队补正。官方预告文章特别强调“仲间越多，组战力越高”。
- **任务/战线适配**：公开攻略资料会按地区任务标注有利组与敌方弱点。势力因此服务大战略选择，而不是纯 UI 分类。
- **收集方向**：玩家为了提升某条战线或解锁成就，会偏向获取特定所属卡。公开 wiki 也按所属列出卡牌。

这套结构的核心价值是：角色收集不会只服务图鉴，每张新卡都能给所属组带来增量价值。

### 4. 属性、数值与成长公式

官方事实：

- AT 是队伍/所属组维度的攻击力，等于该组成员合计加领队补正；战斗行动也按组执行。
- HP 是全前线部队共享耐久，等于已配置队伍合计加领队补正；伤害不是按单个组分别承受。
- 领队角色通过战斗或地图 EXP 获得经验，rank 提升后能力上升。
- 同一张卡重复获得，会获得与 rank 提升不同来源的能力 bonus。
- 勋章 3 个提升 1 点部队 rank；部队 rank 决定可选择的部队奖励数量。

公开资料整理的角色卡公式：

- `当前值 = (基础值 + 基础值 bonus) * (5 + 星 rank)`。
- 星 rank 到达区间阈值时，基础值 bonus 增加。
- 同一卡重复获得会提升“重叠 rank”，公开资料整理为重叠 rank 1 约 +10%，重叠 rank 10 约 +100%。

这些公式说明原作有三条并行成长线：

- **个卡 rank 成长**：通过 EXP 增长，影响卡自身 HP/AT。
- **重复卡成长**：通过再次抽到/拿到同卡，提升重复 bonus。
- **全局部队成长**：通过勋章和部队奖励，解锁配置位、势力使用权、稀有掉落或数值倍率。

本项目不需要照搬公式，但可以借鉴“个体成长 + 重复补偿 + 全局军团成长”的三层结构。

### 5. 技能种类

官方系统页确认：战斗中各领队有战斗技能，技能有 AP 消耗；没有 AP 消耗的技能可能是常时增强、概率触发或特定条件自动触发。

公开 wiki 把技能/效果按大类整理为：

- 物理攻击：近接、远隔等。
- 魔法攻击：元素或特殊攻击。
- 回复行动：HP 回复、异常状态处理、部分再生/反再生相关效果。
- 支援行动：添加支援状态、召唤、增益、AP 回复、持续效果等。
- 妨害行动：打断敌方行动、施加弱体或异常。
- 自动行动 / 被动效果：无需 AP 或满足条件自动触发。
- 状态 UP：编成时或持有时提高战力/属性。
- 状态异常：毒、睡眠、混乱、弱体、诅咒、燃烧、冻结、感电等公开攻略常见状态。

AP 设计的关键不是“每回合能出几张牌”，而是“多名领队共享行动预算”。同一轮里，玩家需要在输出、防御、治疗、打断、蓄 AP、异常控制之间取舍。

### 6. 战斗中的卡牌使用方式

官方系统页给出的战斗流程：

- 战斗是回合/round 制，无实时操作。
- 回合开始时敌方行动已经决定并显示，玩家可以预读。
- 玩家先行动，使用当前 AP 支付各领队技能。
- AP 用完或主动结束后进入敌方行动；敌方按预告行动结算。
- 大招 AP 消耗高，可以攒到后续 round；也可以根据敌方预告选择防御、阻碍或治疗。

编成关系：

- 魔人讨伐队最多由 7 个队伍组成。
- 每个前线配置位放置某所属组的一名领队；领队决定该组前线技能。
- 非领队成员不上技能栏，但计入所属组的合计战力。
- 未编入前线的队伍是后备；官方说明存在条件下的战斗中换队/换领队。

这使卡牌有两种价值：

- **上场价值**：领队技能、AP 曲线、属性、状态处理能力。
- **库存价值**：不当领队也提供所属组 AT/HP 或道具增益。

### 7. 卡牌获取方式

官方事实和公开资料合并后，可以归纳为：

- **剧情/事件固定加入**：部分角色通过任务事件或任务完成固定成为同伴。
- **Quest 奖励**：准备、作战、据点等不同阶段的任务可给固定奖励或触发角色加入。
- **宝箱三选一**：地图宝箱通常出现若干卡/奖励，从中选择一张；有时少于三张。
- **普通池与稀有池**：普通宝箱也可能出高稀有候选，稀有宝箱则保证更高稀有度候选。
- **固定候选宝箱**：有些宝箱的候选大致固定，例如三张中有一张固定。
- **特殊条件卡**：公开 wiki 单独整理固定出现卡与特殊条件卡，说明部分卡受剧情、事件、rank、周回或其他解锁条件影响。
- **重复卡**：重复获得同卡不是转换为货币，而是提升该卡能力。
- **周回/成就相关**：公开攻略与官方支持页都显示成就、图鉴、既读/状态与周回体验有关；公开攻略中也常以周回奖励、初始 rank、部队 rank、稀有运等作为复玩策略。

对设计来说，原作奖励池不是单纯随机抽卡，而是“固定关键节点 + 局部随机选择 + 特殊条件解锁 + 重复补偿”的混合。

### 8. 卡牌系统如何服务大战略

官方系统页把游戏流程拆成准备、作战、据点和战况报告。每个 turn 通常只能选择有限任务；即使任务失败或撤退，时间也会推进。地区战果会降低本 turn 受害，击败关键敌人会永久降低对应地区压力。

卡牌系统在这里承担几项战略功能：

- **把“多战线”转化成“多势力成长”**：想支援某地区，往往需要对应势力的前线战斗力、合适属性与状态处理。
- **让收集直接影响战局承压**：多拿角色不只是补图鉴，而是提高所属组总 AT/HP，从而让后续高压战斗可行。
- **制造不可全拿的选择**：宝箱三选一、任务路线、时间推进、地区压力与死亡/错过事件组合，使玩家不能在单周目自然拿完所有收益。
- **支撑多结局和多周目**：失败、撤退、错过、成就奖励、特殊条件卡和不同路线，让下一周目可用不同优先级重新规划。
- **把剧情角色变成系统资产**：角色事件、食券、固定加入和特殊卡让叙事内容也进入成长经济。

## 对 `rance` 项目的可借鉴设计启发

### 1. 用“角色卡资产”替代纯牌库思维

如果 `rance` 目标是大战略 RPG + 卡牌化角色，卡牌不应只存在于手牌/牌库。更合适的基础定义是：

```ts
type CardVariant = {
  id: string;
  characterId?: string;
  factionId: string;
  role: "character" | "item" | "supportReward";
  rarityTier: "common" | "rare" | "fixed" | "special";
  baseHp: number;
  baseAttack: number;
  rank: number;
  duplicateRank: number;
  skills: SkillId[];
  unlockTags: string[];
};
```

这样一张卡既能被收集、编入队伍、计入势力总战力，也能提供可操作技能或被动资产。

### 2. 区分“领队技能价值”和“库存战力价值”

原作最值得借鉴的是：不上场的同势力角色也有贡献。`rance` 可以让每个势力有：

- `leader`：决定当前战斗技能、行动风格、AP 曲线。
- `members`：提供势力 AT、HP、支援槽、后勤或被动。
- `reserve`：不在本次前线，但仍能贡献低比例后勤值。

这能降低抽到“非主力角色”的挫败感，也让角色池越大越有战略意义。

### 3. 用势力做大战略接口

势力不只是角色标签。建议将势力同时连接：

- 地区任务的有利/不利修正。
- 战斗编成槽。
- 势力人数或势力 rank。
- 战略层收益，例如地区支援、补给、情报、士气。
- 特殊奖励池，例如某地区任务更容易出该势力角色。

### 4. 采用混合奖励池

推荐奖励结构：

- 主线关键角色固定给，避免核心系统被随机卡死。
- 普通宝箱给 3 选 1，制造短期决策。
- 稀有宝箱保证高价值候选，但不保证指定角色。
- 特殊条件卡用成就、事件、角色关系、势力 rank 解锁。
- 重复卡提供数值或专精成长，避免变成纯垃圾。

### 5. 把重复卡设计为“横向专精”

原作重复卡偏向数值 bonus。`rance` 可以保留低成本数值补偿，但不要让重复堆叠成为唯一最优。可考虑：

- `duplicateRank` 提供小额 HP/AT。
- 特定阈值解锁技能微调、皮肤、图鉴条目或势力称号。
- 重复超过上限后转为通用训练资源，但转化率低于直接拿新卡。

### 6. AP 系统优先服务多角色协同

原作 AP 的价值在于多个领队共享预算。`rance` 可以设计：

- 全队共享 AP，而非每张卡单独能量。
- 技能成本分 0/1/2/3+，高费技能需要攒或配合 AP 回复。
- 敌方行动预告公开，鼓励防御、打断、治疗和异常控制。
- 0 AP 技能主要是被动、条件触发或低伤害补位，不要泛滥。

### 7. 建立“错过但不崩盘”的路线系统

原作利用时间推进、地区压力和路线分支制造不可全救。`rance` 可以借鉴压力结构，但需要更清晰的反馈：

- 任务错过会改变地区状态、奖励池、角色可用性。
- 关键死亡/离队应提前给信号，而不是隐藏惩罚。
- 单周目无法全收集，但主线失败不应只来自信息不透明。

### 8. 多周目图鉴应记录“发现路径”

图鉴不只列卡牌。建议记录：

- 首次获得周目、来源任务、来源奖励类型。
- 是否固定、随机、特殊条件或错过后补救。
- 是否已看角色事件。
- 是否已达重复阈值。
- 是否属于某个势力收集成就。

### 9. 全局军团成长独立于个卡等级

原作有部队 rank / 部队奖励。`rance` 可以设计 `ArmyRank`：

- 来源：勋章、地区战果、势力成就、关键任务。
- 用途：解锁前线槽、后备槽、稀有奖励权重、势力上限、支援位。
- 可重置或可切换：让玩家根据战局临时调整，而不是永久点错。

### 10. 战斗数值公式要保持可解释

原作公开资料中的公式非常线性，便于玩家理解“rank、基础值、重复 bonus”为什么提升战力。`rance` 不必使用同一公式，但建议：

- 单卡公式保持 2 到 3 个主要变量。
- 势力总战力公式在 UI 中可追溯到成员贡献。
- 重复、装备、部队奖励分开展示。
- 避免隐藏过多乘区导致玩家无法规划。

## 不建议照搬的部分

- **不要照搬专有角色、国家名、剧情路线、卡名、技能名、图片或完整数据表**。本文只借鉴结构。
- **不要完全复刻成人向内容驱动的角色事件体系**。本项目应只保留“角色事件推动成长/关系/解锁”的结构。
- **不要把大量关键角色锁在无提示随机池**。原作目标是系列终章和高复玩度；新项目早期需要更可控的体验。
- **不要把公式复杂度前置**。先做小规模可解释公式，再扩展稀有、重复和势力 bonus。
- **不要让重复卡形成无限数值膨胀**。重复收益应有软上限或横向用途。
- **不要让单周目不可全救变成信息惩罚**。压力与牺牲要可读、可预期、可复盘。
- **不要让势力分类只做 UI tab**。如果势力不影响战斗、奖励或战略收益，它会退化成筛选器。

## 面向本项目的设计建议

1. 建立 `Character`、`CardVariant`、`Faction`、`Skill`、`RewardPool`、`ArmyRank` 六个基础数据模型，避免把角色身份、卡牌版本和掉落条件混在一个表里。
2. 每张角色卡至少包含：角色 id、卡牌版本 id、势力、rank、HP、AT、技能组、获取来源、重复 rank、图鉴状态。
3. 每个势力维护独立成员池与总战力；上场领队决定技能，未上场成员仍贡献势力战力或后勤值。
4. 战斗采用共享 AP 与敌方预告，技能按攻击、防御、治疗、支援、妨害、被动分类。
5. 奖励池使用“固定关键奖励 + 三选一随机宝箱 + 稀有宝箱 + 特殊条件卡”的混合结构。
6. 重复卡给小额数值和阈值奖励，不建议做无限堆叠；超过上限可转为训练资源或势力经验。
7. 设计 `ArmyRank` 作为全局成长线，用于解锁前线槽、后备槽、稀有权重、特殊势力和支援位。
8. 让地区任务偏向不同势力与属性，使玩家在战略层自然需要多势力培养。
9. 角色死亡、离队、错过与地区失守可以存在，但 UI 必须提前提示风险，并在图鉴/时间线中记录原因。
10. 多周目继承图鉴、成就、部分全局 bonus 和“已知获取线索”，但不要直接继承所有战力，避免破坏新周目规划。
11. 道具/支援类卡应归属势力或后勤系统，成为队伍建设资产，而不是一次性消耗品列表。
12. 第一版原型只实现少量势力、少量卡和一条地区压力线，先验证“收集 -> 势力增强 -> 编成决策 -> 战局推进”的闭环。
